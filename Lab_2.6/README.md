# Farm Management Database

База даних для управління фермерським господарством.  
Реалізована на **Microsoft SQL Server (T-SQL)** в рамках практичних робіт з дисципліни «Бази даних».

---

## Предметна область

На фермі є кілька ділянок землі з унікальними номерами. Кожна ділянка має площу, зрошення та посіяну культуру. Ділянку обслуговує одна бригада, але бригада може обслуговувати кілька ділянок. Фіксуються внесення добрив: яке добриво, на яку ділянку, яким працівником і коли. Кожен працівник може керувати певними видами сільгосптехніки.

**Запити до бази:**
- Визначити площу під кожну культуру
- Вивести робітників бригади разом з технікою
- Знайти культуру з максимальною врожайністю
- Знайти бригади з кількістю техніки понад середню
- Скласти список ділянок з невідповідними добривами

## CURSOR.SQ

Файл містить реалізацію транзакцій, обробки помилок, курсорів та оптимізації запитів.

### Завдання 3 — Транзакція з умовою ROLLBACK
Оновлює суму добрива на 10%, потім перевіряє наявність ділянок площею > 100 га.  
Оскільки таких ділянок немає — виконується **ROLLBACK**, зміни скасовуються.

```sql
BEGIN TRAN;
UPDATE FERTILIZER_APPLICATION SET amount_kg = amount_kg * 1.10 WHERE application_id = 1;
IF (SELECT COUNT(*) FROM LAND_PLOT WHERE area > 100) = 0
    ROLLBACK;
ELSE
    COMMIT;
```

---

### Завдання 4 — Перевірка @@ERROR
Спроба оновити `worker_id = 99999` (якого немає) → порушення FK-обмеження `FK_APPLICATION_WORKER` → `@@ERROR = 547` → **ROLLBACK**.

> `@@ERROR` скидається після кожного наступного оператора, тому перевірку треба виконувати одразу після проблемного оператора.

---

### Завдання 5 — TRY...CATCH
Вставка запису з дублікатом `application_id = 1` → помилка 2627 (порушення PRIMARY KEY) → **ROLLBACK** у блоці CATCH з виведенням `ERROR_MESSAGE()`, `ERROR_NUMBER()`, `ERROR_LINE()`.

> TRY...CATCH надійніший за @@ERROR — не залежить від порядку операторів.

---

### Завдання 6 — Масова вставка 10 000 рядків
Створюється таблиця `FERTILIZER_LOG`. Цикл `WHILE` вставляє 10 000 рядків у єдиній транзакції — або всі, або жоден (атомарність).

```sql
-- Результат перевірки:
SELECT COUNT(*) FROM FERTILIZER_LOG;  -- → 10000
```

### Завдання 7 — Процедура з часовими мітками
Процедура `usp_GetBrigadeSummary` повертає зведення по бригадах. Модифікована: на початку і в кінці фіксується `SYSDATETIME()`, виводиться тривалість у мілісекундах.

```sql
EXEC usp_GetBrigadeSummary;           -- всі бригади
EXEC usp_GetBrigadeSummary @brigade_id = 1;  -- тільки бригада №1
```

**Виведення у Messages:**
```
[START] usp_GetBrigadeSummary  2024-03-10 14:23:01.1234567
[END]   usp_GetBrigadeSummary  2024-03-10 14:23:01.1389234
[TIME]  Тривалість: 15 мс
```

### Завдання 8–9 — Складний запит: без індексу vs з індексом

Запит з'єднує 6 таблиць (`FERTILIZER_APPLICATION`, `LAND_PLOT`, `CROP`, `FERTILIZER`, `WORKER`, `BRIGADE`), фільтрує по даті та `soil_quality_index > 7.0`, групує з `HAVING`, сортує за `total_kg DESC`.

**Індекси (завдання 9):**
```sql
CREATE NONCLUSTERED INDEX IX_FA_Date_Plot_Brigade
ON FERTILIZER_APPLICATION (application_date, plot_id, brigade_id, fertilizer_id, worker_id)
INCLUDE (amount_kg, application_id);

CREATE NONCLUSTERED INDEX IX_LP_PlotId_SoilCrop
ON LAND_PLOT (plot_id, soil_quality_index)
INCLUDE (crop_id, area, brigade_id);
```
Table Scan → **Index Seek**. Економія I/O у 5–20 разів при великих даних.

### Завдання 10 — Cursor 1
Той самий запит реалізований через курсор `FORWARD_ONLY READ_ONLY`. Результат збігається з завданням 8, але виконання повільніше через рядкову (row-by-row) обробку.

---

### Завдання 11 — Cursor 2 (двічі без DEALLOCATE)
Курсор виконується двічі: після першого запуску — `CLOSE` (без `DEALLOCATE`), потім повторно `OPEN`. `DEALLOCATE` — лише після другого запуску. Результати обох запусків ідентичні.

## Висновки

| Порівняння | Результат |
|---|---|
| Індекс vs без індексу | Індекс прибирає Table Scan → суттєве прискорення |
| SET-запит vs курсор | SET-запит у рази швидший для агрегацій |
| CLOSE vs DEALLOCATE | CLOSE зберігає курсор у пам'яті для повторного OPEN |
| Коли курсори доречні | Складна умовна рядкова логіка, ETL, генерація документів |
